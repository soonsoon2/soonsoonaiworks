# UI Template

UI/UX 초기 프로토타이핑용 컴포넌트 샘플 프로젝트다.

## Purpose
- 도구형/편집기형 UI의 기본 컴포넌트 감각을 빠르게 확인한다.
- 실제 제품 구현 전에 밀도, 대비, 토큰, 패널 구조, 상태 표현을 검토한다.
- 유지할 만한 패턴만 프로젝트 지시나 공통 컴포넌트로 승격한다.

## Reuse Model
- 이 프로젝트는 복사해서 쓰는 완제품 템플릿이 아니라, 새 UI 작업의 출발 기준으로 쓴다.
- 새 프로젝트를 만들 때는 레이아웃, 토큰, 컴포넌트 상태를 먼저 비교하고 필요한 패턴만 가져간다.
- 반복해서 쓰는 패턴은 `styles.css`의 토큰, 버튼, 패널, 리스트, 탭, 폼, 테이블, 모달 단위로 분리 후보가 된다.
- 특정 제품 기능이나 도메인 문구는 템플릿으로 승격하지 않는다.
- 반복 사용이 확인된 패턴만 이 폴더의 기준 화면에 남긴다.

## Prototype Flow
1. UI 유형을 정한다: 도구/대시보드/IDE, 마케팅/브랜드, 게임/인터랙티브.
2. 이 템플릿에서 가장 가까운 패널 구조와 컴포넌트 상태를 고른다.
3. 색상, 밀도, 타입 스케일을 새 프로젝트의 토큰으로 바꾼다.
4. 한 화면 또는 한 패널만 먼저 구현한다.
5. 브라우저에서 데스크톱과 모바일 폭을 확인한다.
6. 유지할 패턴만 실제 프로젝트로 옮긴다.

## Listbox Checks
- 네이티브 `select` 팝업이 전체 UI 톤을 깨면 커스텀 리스트박스를 쓴다.
- 공용 리스트는 기본, hover, focus, selected, disabled, empty 상태를 같이 확인한다.
- 행에는 아이콘, 제목, 보조 설명, 상태 배지를 넣어도 줄 높이와 정렬이 흔들리지 않아야 한다.
- 긴 이름, 검색 결과 없음, 모바일 폭, 패널 끝에서 열리는 팝업 위치를 확인한다.
- 키보드로 열기, 위/아래 이동, 선택, 닫기가 가능해야 한다.

## Popup Menu Checks
- 팝업 메뉴는 값을 선택하는 리스트박스와 분리하고, 즉시 실행 명령에만 쓴다.
- 메뉴 안에서 옆으로 다시 열리는 메뉴는 `submenu` 또는 `cascading menu`라고 부른다.
- `New Set`처럼 화면 위에 독립적으로 뜨는 창은 `modal dialog`이며 submenu와 다르게 취급한다.
- 메뉴에는 기본, hover, focus, danger, divider, checked 상태를 같이 확인한다.
- 버튼 메뉴와 컨텍스트 메뉴가 같은 토큰과 행 구조를 공유해야 한다.
- 메뉴는 다른 팝업과 동시에 열리지 않아야 하고, 바깥 클릭과 Escape로 닫혀야 한다.
- 패널 끝이나 모바일 폭에서 메뉴가 화면 밖으로 밀리지 않는지 확인한다.

## Settings Panel Checks
- 좌측 하단 설정은 반투명 backdrop 위에 뜨는 `modal overlay` 또는 `modal dialog`로 본다.
- 설정 모달에는 탭, 토글, 리스트박스, 슬라이더, 상태행, 단일 액션을 함께 확인한다.
- 패널이 열릴 때 기존 메뉴와 리스트박스는 닫혀야 한다.
- 닫기 버튼, backdrop 클릭, Escape로 편집 화면으로 돌아가는지 확인한다.
- 작은 화면에서도 중앙 dialog와 내부 스크롤이 깨지지 않아야 한다.

## Chat Composer Checks
- 좌측 rail의 Chat 버튼에서 열리는 채팅 drawer는 모델 선택, 추론 강도, 파일 첨부, 마이크, 전송 버튼을 한 세트로 확인한다.
- 첨부 칩은 추가, 제거, 긴 파일명, 여러 개 상태를 확인한다.
- 마이크는 녹음 중 상태가 버튼만 봐도 구분되어야 한다.
- 전송 후 입력창 초기화, 메시지 추가, 스레드 스크롤을 확인한다.
- Chat 버튼, 닫기 버튼, Escape로 drawer 열림 상태와 포커스 복귀를 확인한다.
- 좁은 drawer 폭에서 컴포저 액션, 입력창, 전송 버튼이 겹치지 않아야 한다.

## State And Action Checks
- loading, empty, error는 같은 패널 밀도 안에서 전환되어도 높이와 정렬이 크게 흔들리지 않아야 한다.
- confirm dialog는 파괴적 액션에만 쓰고, 취소와 실행 버튼의 위계가 분명해야 한다.
- upload progress는 queued, active, done 상태와 긴 파일명을 같은 행 구조에서 확인한다.
- retry, confirm, progress 버튼은 실제 상태 변화나 toast로 피드백을 확인한다.

## Maintenance
- 공개 리소스에는 정적 앱, README, 실행 방법, 유지/제외 기준만 남긴다.
- 제품별 기록, 실험 로그, 로컬 절대 경로, 비밀 정보는 포함하지 않는다.
- GitHub template repository로 전환할 때는 기본 브랜치 파일 구조가 새 저장소 생성 기준이 된다는 점을 확인한다.

## Run

```sh
cd resources/ui-template
python3 -m http.server 4173
```

Then open `http://localhost:4173/`.

## Keep Or Remove
- Keep: 컴포넌트 밀도와 패널 구조가 반복 작업에 실제로 도움이 될 때.
- Revise: 색상/간격/타입 스케일은 좋지만 화면 구성이 과할 때.
- Remove: 실제 작업 흐름보다 샘플 유지 비용이 커질 때.
